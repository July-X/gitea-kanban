用户贴 Gitea web 截图作为「期望应该长这个样子」参考，对比发现两个不一致：

## 改动 1 · 评审系统事件和 review body 拆成 2 张独立卡

旧版 timelineItems computed 把 review.body 复制到评审事件卡内文，导致对话区
出现「一张卡」（彩色边框 + body 内容）。

Gitea web 的实际行为：review 提交后服务端同时落两条记录：
  - /pulls/{index}/reviews 端点：1 条 review 事件记录
  - /issues/{index}/comments 端点：1 条 CommentTypeReview 类型的 issue comment
    （body 跟 review body 一致，关联到 review.id）

Gitea web UI 把这两条渲染成两张独立卡（视觉上一行系统事件 + 紧跟一行普通评论），
但旧 app timelineItems 把 review.body 直接塞进 review 事件卡，导致：
  - 同一内容在对话区出现两次（如果前端用 ListPullComments + ListPullReviews 都拿）
  - 或者只显示一次但吞掉了 review body（与 Gitea web 不一致）

修复：timelineItems 里 review 事件卡 body 设为空（''）。review body 已经作为
CommentTypeReview 类型的 issue comment 落到 /issues/{index}/comments 端点，
在上面 commentPanels 迭代时作为 source: 'comment' 进时间线，自然会渲染成
独立普通评论卡 + body markdown。修复后对话区显示：

  [kanban_bot 已评论] ← 评审事件卡（无内文，状态徽章）
  [kanban_bot] 再修改一下，沉淀沉淀 😉  ← 普通评论卡（review body）

与 Gitea web UI 严格 1:1 对齐。

回归证据：用户 PR #74 截图 Gitea web 显示「评审 + 留下了一条评论」两张卡，
旧 app 合并显示 1 张卡 + 重复内容。

## 改动 2 · 加合并检查警告区（红框）

对齐 Gitea web 的「此分支已经包含在目标分支中」红框提示（用户截图最底部那条）。

条件：selectedPR.state === 'open' && !selectedPR.mergeable

显示：
  - 红色边框 + GitBranch 图标
  - 主标题：「此分支已经包含在目标分支中，没有什么可以合并。」
  - 折叠的「查看命令行提示」详情，含 git fetch + git checkout 命令
  - 命令里的分支名取自 selectedPR.head.ref
