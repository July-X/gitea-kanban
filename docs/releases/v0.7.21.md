# v0.7.21 —— macOS .app 启动期 gh 探测：PATH 补全 + ResolveGhPath()

> **TL;DR**：Mac 安装版（.app 双击启动）时，用户已安装 gh 但应用找不到（`exec.LookPath("gh")` 失败），因为 launchd 派生的进程 PATH 不含 `/usr/local/bin`（Homebrew 默认路径）。v0.7.21 在 `OnStartup` 扫描常见 gh 路径 + 追加到当前进程 PATH，统一用 `ResolveGhPath()` 探测；同时给 Connect 入口也换上 `gh_not_installed` 错误码，前端 toast 引导安装页一致。

## 改动详情

### 根因：macOS .app launchd 派生时 PATH 截断

**症状**：Mac 用户从 .dmg 安装应用后，双击启动 → GitHub 仓库同步报 `gh_not_installed`，但终端 `which gh` 正确返回 `/usr/local/bin/gh`。

**根因**：macOS 双击 `.app` 或从 Launchpad 启动时，进程由 **launchd** 派生，继承的 `PATH` 通常只含 `/usr/bin:/usr/sbin:/bin:/sbin`。`/usr/local/bin`（Homebrew Intel）、`/opt/homebrew/bin`（Homebrew Apple Silicon）等用户扩展路径都不在。而用户在终端跑 `which gh` 时，zsh 已加载 `~/.zshrc` 扩展了 PATH。

用户之前把 gh 装在 `/usr/local/bin/gh`，.v0.7.20 加的 `exec.LookPath("gh")` 在 .app 启动时看不到这个路径 → 仍报 `gh_not_installed`。

### P0 修复：启动期 PATH 补全

**设计**：

1. `gitbinary.EnsureGhInPath()`：启动期扫描常见 gh 安装路径（macOS: `/opt/homebrew/bin` / `/usr/local/bin` / `/opt/local/bin` / `~/.local/bin` / `~/.asdf/shims` / mise 安装目录；Linux: `/usr/local/bin` / `/snap/bin` / `~/.local/bin` / `/opt/gh/bin`），把第一个找到的目录追加到当前进程 PATH（`os.Setenv`）
2. `gitbinary.ResolveGhPath()`：统一探测入口 — 用户覆盖 → `exec.LookPath`（PATH 已被补过） → 返回 `*GhNotFoundError`
3. `app.go` `OnStartup` 在 `gitbinary.Init` 之后调一次（一次性，不递归）
4. `native.go` 的 `CloneWithFilter` / `FetchWithFilter` 改调 `ResolveGhPath()`
5. `app_auth.go` 的 GitHub 入口错误码统一为 `CodeGhNotInstalled`（前端 `buildGhInstallToastError` 覆盖所有入口）

**为什么不在 `clone.go` / `sync.go` 降级到 go-git**：用户已声明用方案 B（优先解决探测问题），且用户本机已装 gh，此举不必要。

**风险与回滚**：
- 子进程 PATH 膨胀：仅追加找到的第一个 gh 目录，长度可控
- nix-shell / 多版本管理器：若 asdf/mise 的 gh 被找到但实际不工作 → 用户可在 SettingsView 手动指定路径（`SetGhOverride`）
- 回滚 = 删除 `EnsureGhInPath()` 调用，保留 `ResolveGhPath()` 的用户覆盖入口

### 不做清单

- 不做内嵌 gh CLI 二进制（方案 B 已定）
- 不做 SettingsView gh 二进制卡片（v0.7.21 只做 P0 临时方案；卡片 UI 留给后续版本）
- 不做 gh 自动 download + install（复杂度高，需 gh 发版 JSON parsing + 跨平台 in-place update）
