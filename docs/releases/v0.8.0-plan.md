# v0.8.0 —— 应用自动更新（参考 DeepSeek-Reasonix 桌面端自更新设计）

> **TL;DR**：用户每次拿到新功能/修 bug 都得手动下载新版本太麻烦。v0.8.0 参考 [DeepSeek-Reasonix](https://github.com/esengine/DeepSeek-Reasonix) 桌面端的成熟自更新架构，为 gitea-kanban 引入 **manifest 拉取 → 多源 fallback → 签名校验 → 断点续传下载 → 分平台安装** 全链路自动更新能力。支持 **stable 单通道** 和 **Windows / macOS 双平台**，Windows 走 in-place 二进制替换（写相邻 new.exe + 重启后 move 覆盖），macOS 走 .app 内 binary 替换 + relaunch**（未签名 build 先走手动下载页兜底，后续加签名/notarize 后开 in-place）。

## 范围与目标

### 核心用户故事
1. 用户启动 gitea-kanban 时自动检查更新（可关闭），有新版本则在 AppShell 顶部显示「发现新版本 vX.Y.Z」banner
2. 用户点击「下载」后带真实进度条（百分比），后台下载并校验签名
3. 下载完成后显示「重启以安装」，点击后 gitea-kanban 自行替换二进制并重启
4. macOS 用户（未签名 build）看到的是「前往下载页」而不是自动下载（跟 Reasonix 路线一致：签名后开 in-place）
5. 设置页「关于」卡片显示当前版本 + 「检查更新」按钮，跟 banner 共用同一个 updater state machine

### 平台差异（重点）

| 平台 | 更新策略 | 签名要求 | 备注 |
|---|---|---|---|
| **Windows** | in-place 二进制替换：写相邻 `gitea-kanban-new.exe` → 写 `.bak` 备份 → 写 `restart-helper.cmd` → exit → 脚本自动 move 覆盖 + relaunch | Authenticode（先跳过） | 单 binary 无 CGO，可直接 in-place（Reasonix 走 NSIS 是因为 CGO/WebKit 架构） |
| **macOS** | 未签名 build 走手动下载页（`OpenDownloadPage()`）；签名+notarize 后改 in-place 替换 `.app/Contents/MacOS/gitea-kanban` + relaunch | Developer ID + Notarization（先跳过） | 跟 Reasonix macOS 路线完全一致：先手动兜底，签名后开 in-place |

### 设计目标
- **零配置**：首次启动即有检查更新能力；CDN fallback 策略跟 Reasonix 看齐（可自建 GitHub release 作首源）
- **不可破环**：签名校验失败 → 删除已下载缓存，绝不在失败签名下 apply
- **可回滚**：update 前备份当前 binary 到 `.bak`，签名校验失败 / apply 失败时自动还原
- **零术语**：UI 文案全中文，"update""manifest""minisign"等内部术语不出现在 UI
- **可测试**：HTTP 端点用 httptest mock，签名用临时 key pair，CI 单测全通过

### 不在本期范围（等后续版本）
- Windows Authenticode 签名 / macOS Notarization 证书流程
- canary 多通道 / Linux 平台支持
- 差量级热更新（block-level patch）
- 多 CDN mirror（R2 等）—— 直接用 GitHub release 作源即可
- Electron 时代的 `electron-updater`（本项目是 Wails + Go binary，不适用）
- 引入 `minisign` 等新依赖（ed25519 走 Go 标准库 `crypto/ed25519`，零新三方依赖）

## 参考架构（DeepSeek-Reasonix desktop auto-updater）

~/2026/code/DeepSeek-Reasonix/desktop/updater.go + updater_app.go 的核心设计：

```
前端 React useUpdater hook
    ↕  Wails bindings （CheckUpdate / DownloadUpdate / InstallUpdate / OpenDownloadPage）
    ↕  Wails EventsEmit("updater:progress" {phase, received, total, err})
后端 updater.go
    ├── manifest fetch：R2 CDN → crash gateway → GitHub fallback（本次只用 GitHub）
    ├── semver 版本比较（dev build 不提示）
    ├── 签名校验（ed25519 — Go 标准库 crypto/ed25519）
    ├── 断点续传下载（Range header + IPv4 fallback）
    ├── 本地缓存（${cacheDir}/updates/downloaded.json）
    ├── per-platform apply（windows→in-place replace+restart helper / mac→签名后 in-place or 手动下载页）
    └── relaunchThroughGuard()
```

gitea-kanban 保留 Reasonix 的核心思路，简化部署链路：
- 用 **GitHub release**（本仓库 `release` 页面）作为 manifest + 安装包唯一源（公司/个人用户都能用）
- manifest 直接取 `https://api.github.com/repos/{owner}/{repo}/releases/latest`（返semver tag + assets 列表）
- 签名方案：**ed25519**（Go 标准库 `crypto/ed25519`，零新依赖；`< 30` 行代码即可集成 Verify，比 minisign 简洁；公钥硬编码，私钥存 CI secret）

## Phase 1：后端 — updater 核心（Go 新增 2 文件）

### 1.1 `app/updater/manifest.go`

**职责**：定义 Manifest / Asset DTO、平台 key 规则、ed25519 公钥常量。

```go
package updater

// Manifest 对齐 GitHub release API 的单层扁平结构
type Manifest struct {
    Version string            // semver，例如 "v0.8.0"
    Notes   string            // release body 文本（markdown）
    PubDate string            // RFC3339
    Assets  map[string]Asset  // key = PlatformKey(GOOS, GOARCH)
}

type Asset struct {
    URL    string // GitHub release browser_download_url
    Size   int64
    SHA256 string // 可选二次校验
    SigURL string // 分离的 .sig 文件 URL（ed25519 detached signature）
}

func PlatformKey(goos, goarch string) string { return goos + "-" + goarch }
func CurrentPlatform() string                 { return PlatformKey(runtime.GOOS, runtime.GOARCH) }
```

**新增依赖**：`crypto/ed25519` + `crypto/x509` + `encoding/base64`（全部 Go 标准库）
**ed25519 公钥**：硬编码在 `verify.go` 常量中（公钥安全可公开，私钥存 CI secret）

### 1.2 `app/updater/updater.go`

**职责**：版本比较、HTTP fetch（3 次 retry + IPv4/IPv6 fallback）、热更新缓存目录、签名校验。核心函数签名：

```go
func (a *App) CheckUpdate() (*UpdateInfo, error)
func (a *App) DownloadUpdate() (*UpdateDownloadResult, error)
func (a *App) InstallUpdate() error
func (a *App) OpenDownloadPage() // 打开浏览器到 release 页
func (a *App) ApplyUpdate()      // convenience：Download+Install 一体
```

**关键子步骤**：
1. `resolveLatestRelease()` 调 GitHub `/releases/latest` 端点 → 解析 tag_name + assets → `Manifest`
2. `normalizeVersion()` 规范化 semver（dev build 返 ok=false）
3. `evaluate()` 比较 running vs latest → `UpdateInfo{ Available, Current, Latest, ... }`
4. `downloadVerify()`：流式下载（Range 续传 + retry）→ 下载 `.sig` → ed25519 校验 → SHA256 二次校验
5. `saveCachedUpdate()` 原子写 tmp+rename 到 `${cacheDir}/updates/`
6. `applyXXX()` 分平台：
   - **Windows**：os.Executable() 拿当前 exe 路径 → 写 `gitea-kanban-new.exe` 到同目录 → 写 `gitea-kanban.bak` 备份 → 写 `restart-helper.cmd`（内容：`/c move /Y new.exe gitea-kanban.exe & start gitea-kanban.exe`）→ 执行 helper → `os.Exit(0)` → helper 负责 move 覆盖 + relaunch
   - **macOS**：未签名 build → `OpenDownloadPage()`（不调 apply）；签名+notarize → 拿 `.app/Contents/MacOS/` 路径 → 写新 binary 到同目录 `.new` → rename 替换 → `exec.Command(os.Executable())` relaunch → `os.Exit(0)`

### 1.3 `app/updater/updater_test.go`

**测试覆盖**（12 个 Go 单测，双平台差异化覆盖）：
- `TestNormalizeVersion`：dev→false / "v0.8.0"→ok / "0.8.0"→补 v
- `TestEvaluate`：running=latest→Available=false / running<latest→Available=true
- `TestFetchManifestMock`：httptest mock GitHub release API → 正确解析
- `TestDownloadVerifyMock`：httptest mock asset+sig URL → 成功 / 篡改 sig → 失败
- `TestApplyWindowsMock`：写 `new.exe` + `.bak` + `restart-helper.cmd` → exit → 验证文件就位
- `TestApplyMacOSSkipped`：未签名 macOS 走 `a.OpenDownloadPage()`（不调 apply）
- `TestApplyMacOSSignedMock`：签名+notarize 后 macOS 走 in-place replace + relaunch
- `TestCanSelfUpdate`：GOOS=darwin→false（未签名）/ windows→true / mac 签名后=true
- `TestPlatformKey`：对齐 key 格式 `windows-amd64` / `darwin-arm64`
- `TestManualUpdateReason`：macOS 未签名时返可读原因字符串
- `TestRetryOnTransientError`：httptest 第一次 500 → 第二次 200
- `TestCacheInvalidOnChannelOrPlatformChange`：缓存失效逻辑

## Phase 5：签名工具 + CI 发版流程

### 5.1 `cmd/sign`（小型 Go CLI，对标 Reasonix `desktop/cmd/sign`）

**职责**（Ph5 后由 CI 或维护者手动调用）：
1. 生成 ed25519 key pair（首次执行）
2. 对 release asset 列表逐个计算 SHA256 + detached signature（`.sig` 文件）
3. 公钥写入 `app/updater/verify.go` 常量

**工作流**：
```bash
# 1. 生成 key pair（一次性，private key 加密存 CI secret）
go run ./cmd/sign genkey ./keys
# 输出：gitea-kanban-public.pem（嵌入 verify.go）+ gitea-kanban-private.pem（加密存 secret）

# 2. 对发版产物签名
go run ./cmd/sign sign ./build/bin/gitea-kanban-* --key ./keys/private.pem
# 输出：gitea-kanban-*.sig
```

### 5.2 CI 流程改造（`.github/workflows/build.yml`）

**现状**：build.yml 跑 macos-14 / windows-latest，但未做签名或 manifest 生成。

**新增 job `publish-release`**（仅在 tag push `v*` 时触发）：
1. 跑 `go test ./app/...`（已有）
2. 跑 `wails build -platform <plat> -o gitea-kanban-<plat>`（已有部分）
3. 跑 `go run ./cmd/sign sign` 对所有 assets 签名
4. 跑 `go run ./cmd/sign manifest` 生成 `latest.json`（返版本 + assets 列表）
5. `gh release create vX.Y.Z` 把所有 assets + latest.json + *.sig 打包上传

> **CI secret 规划**：新增 `SIGN_PRIVATE_KEY`（ed25519 private key base64）、`SIGN_PASSWORD`（加密私钥用）。公钥硬编码在代码里。
> **平台矩阵**：仅 `macos-14`（darwin/universal）+ `windows-latest`（windows/amd64），无 Linux job。

### 5.3 tag 命名约定

gitea-kanban 现有 tag 命名是 `v0.7.19`（无 `desktop-` 前缀，因为 gitea-kanban 是单一产品线）。继续沿用 `v*` → manifest tool 读 `v*` tag_name → 直接传给 `evaluate()`。

**触发 condition**：
```yaml
on:
  push:
    tags: ["v*"]   # 跟 CLI 不同，gitea-kanban 单产品直接用 v*
```

## Phase 6：应用数据目录 + 缓存 + 偏好

### 6.1 缓存目录（`${dataDir}/updates/`）

**对齐 AGENTS §8.2**：缓存态数据放 `${dataDir}`（即 `~/.gitea-kanban/`）下：

```
~/.gitea-kanban/
├── updates/
│   ├── gitea-kanban-v0.8.0-windows-amd64.exe  ← Windows 二进制缓存
│   ├── gitea-kanban-v0.8.0-darwin-arm64       ← macOS binary 缓存（.app 提取）
│   └── downloaded.json                        ← 元信息（version/channel/path/sha256）
├── state.json
├── logs/
└── workspace/
```

**downloaded.json 结构**：
```json
{
  "version": "v0.8.0",
  "channel": "stable",
  "platform": "windows-amd64",
  "path": "~/.gitea-kanban/updates/gitea-kanban-v0.8.0-windows-amd64.exe",
  "size": 23456789,
  "sha256": "abc123...",
  "downloadedAt": "2026-07-20T10:30:00Z"
}
```

**缓存失效**：
- Channel 或 platform 不匹配 → 失效
- 文件不存在 / SHA256 校验失败 → 删除并重下
- 版本等于当前 running 版本 → 清理缓存

### 6.2 偏好读写

`localStore.prefs["app.checkUpdates"]`（bool，默认 true） → 启动时是否自动检查。SettingsView toggle 通过 store action 修改。

```ts
// 前端读
const checkUpdates = store.prefs["app.checkUpdates"] ?? true
// 前端写
store.setPref("app.checkUpdates", !checkUpdates)
```

**缺 store action 时补充**：store 包新增 `SetPref(key string, value any)` action。

## Phase 7：安全 + 错误处理 + 可观测性

### 7.1 安全层

| 攻击向量 | 防御 |
|---|---|
| 中间人替换安装包 | ed25519 签名校验（私钥不在 CI 外泄） |
| 降级到旧版本 | manifest running > latest 时拒绝 apply（需 `force` 参数） |
| 缓存投毒 | SHA256 二次校验 + 签名二次校验（sign → SHA256 双保险） |
| 断点续传 Range 欺骗 | HTTP 206 校验 Content-Range 头 + 总量对齐 manifest asset.Size |
| 私钥泄露 | CI secret 托管，代码库只放公钥；签名在 CI 内部私下执行 |

### 7.2 错误分类（对齐 `IpcError` 三件套）

前端 `UpdateStatus.kind === "error"` 时显示 `message` + 重试按钮。

| 错误码 | 含义 | 处理 |
|---|---|---|
| `update:manifest` | GitHub API 4xx/5xx 或网络错误 | 「检查网络后重试」 |
| `update:signature` | 签名校验失败 | 「安装包可能被篡改，请重新下载」 |
| `update:sha256` | SHA256 不匹配 | 「下载不完整，请重试」 |
| `update:platform` | 当前平台无对应 asset（如用户在不受支持的 OS 上） | 「当前平台暂不支持自动更新，请手动下载」 |
| `update:permission` | 写文件权限不足（Windows UAC / macOS 沙箱） | 「无替换权限，请以管理员身份运行或手动更新」 |
| `update:codesign` | macOS Gatekeeper 拦截未签名的新 binary | macOS 未签名 build 的通用提示，引导「前往下载页」 |

### 7.3 日志（slog）

`updater.go` 所有关键节点写 slog.Info/Warn/Error：
- 检查成功：`"update check" version=0.7.19 latest=0.8.0 available=true`
- 下载进度：`"update download" version=0.8.0 received=23456789 total=45678901`
- 签名校验：`"update verify" ok=true version=0.8.0`
- apply 成功：`"update apply" from=0.7.19 to=0.8.0`
- 错误：`"update failed" err=... code=update:signature`

> 禁止在日志里打 URL query string / token（如 GitHub API URL 本身是公开的，但习惯仍避免）。

## 风险与已知问题

| 风险 | 缓解 |
|---|---|
| GitHub API 速率限制（未认证 60/h） | 启动检查一天只跑 1 次（localStorage 看 lastCheckAt）+ 带 ETag 省请求 |
| macOS Gatekeeper 拦截替换后的新 binary | 仅在签名版启用 in-place；未签名版手动下载页兜底 |
| Windows 文件锁（应用运行时无法替换自己的 exe） | `restart-helper.cmd` 方案：exit 后脚本 move 覆盖 + relaunch（避开文件锁） |
| Windows WebView2 runtime 跟 updater 的Rename 冲突 | 实测验证：updater 在 rename 时已释放 WebView2 句柄（os.Exit 后进程已死，无冲突） |
| 跨大版本（如 v0.7.x → v0.8.0）数据兼容性 | state.json schemaVersion 机制（已有）处理 |
| ed25519 私钥泄露 | CI secret 托管 + 启动检查仅拉公钥校验（公钥硬编码而非网络拉） |
| macOS .app 包路径解析失败（如用户手动移动 .app 位置） | 用 `os.Executable()` 拿真实路径，不硬编码 `/Applications/` |
