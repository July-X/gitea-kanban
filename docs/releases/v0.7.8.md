# v0.7.8 —— push / merge 事件详情 根因修复

> **TL;DR**：v0.7.7 修了 push 事件 commit 列表 + merge 事件 commit 链接，但 user 反馈"对话事件中的 commit 信息还是看不到" —— 实际跑出来发现 push/merge 事件**几乎都不渲染**。根因不在模板，v0.7.7 **写代码时没实测过 Gitea 1.26+ API**，凭印象假设了 4 个独立顶层字段 + type 字符串"push"/"merge"，实际 Gitea 端点是 snake_case `pull_push`/`merge_pull` + 真实 commit 信息在 body JSON 字符串里。v0.7.8 重新对齐 Gitea 端实际行为。

## 定位

v0.7.8 是 **v0.7.7 根因修复** —— 之前 v0.7.5 + v0.7.7 写的 push/merge 事件逻辑**全部不工作**（type 字符串不匹配 + 4 个独立字段 API 不返回 + commitsByPR 双 Map + 错 key），user 看到的还是"系统事件 dot + 头像 + 时间 + 空 verb"。Gitea web 显示的"X 推送了 1 个提交"+"commit_ids 列表"+ "合并 commit {sha} 到 main"在我们 app 全部不显示。

**v0.7.8 唯一目的：把 push / merge 事件在 Gitea 1.26+ API 下能正确渲染出来**。

## 根因（user 反馈 ⑦）

实测 Gitea 1.26.2 `/api/v1/repos/{owner}/{repo}/issues/{index}/timeline` 端点返回的 push / merge 事件实际结构：

```json
// type=pull_push 事件（Gitea 1.24+ snake_case 命名）
{
  "id": 372,
  "type": "pull_push",                          // ← 实际是 "pull_push" 不是 "push"
  "body": "{\"is_force_push\":false,\"commit_ids\":[\"54244e...\"]}",  // ← 真实数据在 body JSON
  "user": {...},
  "created_at": "2026-06-14T19:27:25+08:00"
  // 没有 old_commit_id / new_commit_id / commits_num / is_force_push 顶层字段
}

// type=merge_pull 事件
{
  "id": 374,
  "type": "merge_pull",                         // ← 实际是 "merge_pull" 不是 "merge"
  "body": "",                                    // ← body 是空字符串
  "user": {...},
  "created_at": "2026-06-14T19:27:29+08:00"
  // 没有 merge commit SHA 字段
}
```

**v0.7.7 假设**（全错）：
1. ❌ type 字符串 = `"push"` / `"merge"` —— 实际是 `"pull_push"` / `"merge_pull"`
2. ❌ push 事件顶层有 `old_commit_id / new_commit_id / commits_num / is_force_push` 4 个独立字段 —— 实际全在 body JSON 字符串里
3. ❌ merge 事件 body 格式固定 `"merged commit {sha} into {branch}"` —— 实际是空字符串
4. ❌ merge commit SHA 从 timeline body regex 抠 —— 实际只能从 PR 详情端点 `merge_commit_sha` 拿

**v0.7.7 副作用**（v0.7.8 也修了）：
- 组件本地 `commitsByPR` ref 跟 store `pull.commitsByPR` 是两个独立 Map，`loadCommits` 写组件本地，`pushEventCommits(item)` 读 store —— 永远空
- `pushEventCommits` key 用错（`activeRepo.value.id` 是仓库 ID，应该是 `pr.index`）
- v0.7.7 加的 `PullDetailDTO.MergeCommitSHA` 字段没人填（Gitea `giteaPullRaw` 漏 raw struct 字段 + 漏 mapping）

## 修复（v0.7.8 全部重写）

### 后端

1. **类型归一化**（`giteaTimelineToItem`）：适配层把 Gitea 端 `pull_push` → `push`、`merge_pull` → `merge`，前端 type 字符串表保持稳定（`systemEventVerb / systemEventIcon / systemEventColor / 模板`全不用改）
2. **解析 push event body JSON**：拿到 `commit_ids` 数组（`CommitIDs []string`）+ `is_force_push` 标志（`IsForcePush bool`），body 解析失败静默忽略（兼容老 Gitea 文本格式）
3. **删 4 个无用字段**：`OldCommit / NewCommit / CommitsNum`（Gitea API 不返回）
4. **补 giteaPullRaw + giteaPullToDetail 映射 `merge_commit_sha`**：PR 详情端点 `/pulls/{index}` 返回的 merge commit SHA 现在正确映射到 `PullDetailDTO.MergeCommitSHA`
5. **GitHub adapter 同源修复**：`githubPullRaw` 加 `MergeCommitSHA` 字段 + `githubPullToDetail` 映射
6. **3 个新测试**（`adapter_test.go`）：
   - `TestGiteaAdapter_ListPullTimeline_PushMergeTypeNormalization` —— 验证 type 归一化 + commitIds 提取 + force push 提取
   - `TestGiteaAdapter_ListPullTimeline_PushBodyInvalid` —— 验证 body 不是 JSON 时静默忽略
   - `TestGiteaAdapter_GetPull_MergeCommitSHA` —— 验证 PR 详情 merge_commit_sha 字段映射

### 前端

1. **删 v0.7.7 错路径**：
   - 删组件本地 `commitsByPR / commitsLoading / loadCommits`（双 Map bug）
   - 删 `pushEventCommits(item)` helper（key 错 + Map 错位）
   - 删 `fullMergeSha(item) / mergeCommitSha(item)` helper（body regex 抠永远 null）
2. **`loadComments` 改为并行 3 个请求**：`fetchPullDetail`（拿 mergeCommitSha）+ `fetchTimeline`（拿 timeline）+ `loadCommits`（拿全量 commit 缓存，commitDetails 补 subject/author 用）
3. **push event 渲染直接用 `item.commitIds`**：后端 giteaTimelineToItem 解析 body JSON 拿到的 SHA 数组，前端 v-for 渲染（每行：GitCommit icon + 短 SHA 链接 + 可选 subject/author 从 store 缓存匹配）
4. **merge event 渲染用 `selectedPR.value?.mergeCommitSha`**：PR 详情端点字段，inline 块显示 "到 {base.ref} + 7 位短码链接"
5. **`store.fetchPullDetail(p)` 新增**：调 `pullsGet` IPC（Wails 绑 `GetPull`），拿到 detail 后浅 patch 到 `currentSelectedItem` + `items` 里对应 PR（mergeCommitSha 字段）
6. **`PullDto` 加 `mergeCommitSha?: string` 字段**：merge event 模板用得到的字段
7. **`store.commitsLoading` export**：tab loading indicator 之前组件本地有，v0.7.8 统一到 store

## 影响

**修好的**：
1. push 事件现在显示完整 commit 列表（每行短 SHA 链接 + 提交消息 + 提交者，仿 Gitea web `commits_list_small` 模板）
2. push event 头部 inline 块显示"head commit 短码链接 + (强制推送) 提示"
3. push event 系统事件 verb 现在显示 "推送了 N 个提交"（带"了"，对齐 Gitea web `push_commits_1` / `push_commits_n` 中文 locale + TrN 单复数）
4. merge event 显示完整"到 {base.ref} + 7 位短码链接"（对齐 Gitea web `templates/repo/issue/view_content/comments.tmpl` "merged commit XXXXXXX into main" 渲染）
5. Gitea 1.26+ 端点实际行为对齐（v0.7.7 之前没实测过 API）

**降级 / 限制**：
- 老 Gitea 版本（<= 1.25）push event body 是文本格式 "added N commits {time}"，type 字符串是 "push" —— v0.7.8 兼容（type 字符串是 "push" 时不进 body JSON 解析分支，留空 commitIds 走 '推送了新提交' fallback）
- commit 列表的 subject / author 从 `/pulls/{index}/commits` 端点拉，按 SHA 短码二次匹配（store loadCommits 缓存里找）—— 找不到时只显示 SHA（API 限制）

## 已知 limitation

- **Gitea 1.26+ merge_pull event body 是空字符串**：merge commit SHA 只能在 PR 详情端点拿，所以 store 需要 `fetchPullDetail` 二次拉取。已实现，但失败时（API 限流 / token 过期）merge 事件 SHA 链接不显示，verb 仍正常显示。
- **Gitea 1.26+ 不返 OldCommit / NewCommit**：force push compare 链接无法精确渲染（只能显示 "(强制推送)" 文字），等 Gitea 端加字段。
- **commit 消息 / 提交者不在 commit_ids 数组里**：Gitea timeline API 只返 SHA，不返每条 commit 的详情，客户端需要二次拉 `/pulls/{index}/commits` 补（v0.7.8 做了，找不到时降级）。
