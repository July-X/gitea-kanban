# v0.8.0 —— 应用自动更新全链路

> **TL;DR**：用户每次拿到新功能/修 bug 都得手动下载新版本太麻烦。v0.8.0 参考 [DeepSeek-Reasonix](https://github.com/esengine/DeepSeek-Reasonix) 桌面端的成熟自更新架构，为 gitea-kanban 引入 **manifest 拉取 → 多源 fallback → ed25519 签名校验 → 断点续传下载 → 分平台安装** 全链路自动更新能力。支持 **stable 单通道** 和 **Windows / macOS 双平台**。Windows 走 in-place 二进制替换（写相邻 new.exe + 重启后 move 覆盖），macOS 未签名 build 走手动下载页兜底。**明文排除 Linux / canary 通道。**

## 改动详情

### 平台差异

| 平台 | 更新策略 | 签名要求 | 备注 |
|---|---|---|---|
| **Windows** | in-place 二进制替换：写相邻 `gitea-kanban-new.exe` → 写 `.bak` 备份 → 写 `restart-helper.cmd` → exit → 脚本自动 move 覆盖 + relaunch | Authenticode（先跳过） | 单 binary 无 CGO，可直接 in-place |
| **macOS（未签名）** | 手动下载页（`OpenDownloadPage()`） | Developer ID + Notarization（先跳过） | 跟 Reasonix macOS 路线完全一致：先手动兜底，签名后开 in-place |
| **Linux** | ❌ 明文排除 | — | 用户拍板 v0.8.0 只 Windows + macOS |
| **canary** | ❌ 明文排除 | — | stable 单通道 |

### 后端实现

**manifest 拉取**：`Check()` 调 GitHub releases latest API（`Accept: application/vnd.github+json` header），提取 tag → `CompareSemver` 判断是否有更新。3 次重试 + 指数 backoff（1s / 2s / 4s）。

**断点续传下载**：`http.Range` 支持，下载过程记录 `received` bytes，失败时 `Resume` 从断点继续。

**本地缓存**：download 产物写 `${cacheDir}/updates/`。apply 前签名校验失败则删除缓存，绝不在失败签名下 apply。update 前备份当前 binary 到 `.bak`，apply 失败时自动还原。

**ed25519 签名校验**：Go 标准库 `crypto/ed25519`（零新依赖）。`VerifySignature(manifest, sig, pub)` 逻辑 `< 30` 行。公钥硬编码在 `app/updater/verify.go` 的 `PublicKeyB64` 常量，**当前是占位符**，release 前必须替换（见「发布前事项」）。

**Wails 进度事件**：`Download` 内部 `onProgress(received, total)` callback 通过 `wruntime.EventsEmit(ctx, "updater:progress", UpdateProgress{...})` 推流，前端 `useUpdate.ts` 订阅。

**启动异步检查**：`OnStartup` 末尾 `initUpdater()` 初始化 Updater（注入 publicKey + cacheDir + logger），然后 `go a.checkUpdatesAtStartup()` fire-and-forget 异步查更新（不阻塞 UI）。

### 前端实现

**singleton 状态机**（`useUpdate.ts`，10 种状态）：

```
idle → checking → available → downloading → downloaded → installing
                    ↓              ↓              ↓
                 error ←←←←←←←←←←←←←←←←←←←←←←←←←←←←
                    ↓
                  manual（macOS 未签名版本）
```

**顶部 UpdateBanner**：`AppShell.vue` onMounted `void check()` 触发检查，有更新时 banner 滑入。5 种文案状态：
- checking：静默，不显示 banner
- available：「发现新版本 vX.Y.Z」
- downloading：进度条 + 已下载 / 总大小
- downloaded：「新版本已就绪」
- manual（macOS 未签名）：「macOS 未签名 → 前往下载页」
- error：「检查更新失败，稍后重试」

**下载进度**：订阅 `updater:progress` 事件，`downloaded` / `total` → `progress`（0-100）。

**「稍后提醒」**：banner dismiss 后存 `checkUpdatesPref = false`，下次 OnStartup 不自动查（用户点「检查更新」才重新触发）。

**网络失败静默降级**：`checkUpdate()` 失败 toast 提示，不 block UI，下次启动再试。

## 发版工具与 CI

### 签名工具用法

```bash
# 1. 生成密钥对（maintainer 本地一次性）
go run ./cmd/sign genkey ./keys --password "$pwd"

# 2. 签名安装包（CI 调用）
go run ./cmd/sign sign \
  --key keys/gitea-kanban-private.b64 \
  --password "$pwd" \
  build/release/gitea-kanban-v0.8.0-macos-amd64.zip

# 3. 生成 manifest（CI 调用）
go run ./cmd/sign manifest \
  --version v0.8.0 \
  --notes docs/releases/v0.8.0.md \
  --repo July-X/gitea-kanban \
  build/release/gitea-kanban-v0.8.0-macos-amd64.zip \
  build/release/gitea-kanban-v0.8.0-windows-amd64.exe
```

`cmd/sign` 支持 3 个子命令：`genkey`（生成 ed25519 密钥对，私钥 argon2id 加密）/`sign`（ed25519 签名）/ `manifest`（生成 latest.json）。

### GitHub Actions 打包矩阵

| Runner | 平台 | 架构 | 产物 |
|---|---|---|---|
| `macos-14` | darwin | amd64 | `gitea-kanban-v*.dmg` |
| `windows-latest` | windows | amd64 | `gitea-kanban-v*.exe` |

触发条件：`git push` tag `v*`。

每个 build job 产出：`.app` bundle（macOS）或 `.exe`（Windows）。

`sign · manifest · publish` job（ubuntu-22.04）：下载 artifact → `scripts/release.sh` 签名 → `gh release create` 发布。

产物命名：`gitea-kanban-${TAG}-macos-amd64.dmg` / `gitea-kanban-${TAG}-windows-amd64.exe`。

### 产物结构

每个 release 包含（8 个文件）：

```
gitea-kanban-v0.8.0-macos-amd64.dmg         # macOS 安装包
gitea-kanban-v0.8.0-macos-amd64.dmg.sig    # ed25519 detached signature
gitea-kanban-v0.8.0-windows-amd64.exe       # Windows 安装包
gitea-kanban-v0.8.0-windows-amd64.exe.sig  # ed25519 detached signature
latest.json                                    # 更新 manifest（含 assets + signature）
latest.json.sig                               # manifest 自身签名
```

## 发布前事项（Maintainer Checklist）

> **⚠️ 正式发版前必须完成，否则自动更新功能无法工作**

### 1. 替换 `PublicKeyB64` 占位符

```bash
# 1. 生成密钥对（维护者本地一次性）
go run ./cmd/sign genkey ./keys --password "$your_strong_password"

# 2. 从输出拿公钥（gitea-kanban-public.b64 的 base64 解码内容）
base64 -d keys/gitea-kanban-public.b64 | xxd | head

# 3. 把公钥的 base64 字符串粘到 app/updater/verify.go
#    替换 PublicKeyB64 = "PLACEHOLDER_REPLACE_WITH_BASE64_ED25519_PUBLIC_KEY_BEFORE_RELEASE"
```

`PublicKeyB64` 当前值：`"PLACEHOLDER_REPLACE_WITH_BASE64_ED25519_PUBLIC_KEY_BEFORE_RELEASE"`。不替换则 `VerifySignature` 永远失败。

### 2. 配置 GitHub Secrets

仓库 Settings → Secrets and variables → Actions 新增：

| Secret 名 | 内容 |
|---|---|
| `SIGN_PRIVATE_KEY` | `keys/gitea-kanban-private.b64` 文件内容的 base64 字符串（一行） |
| `SIGN_PASSWORD` | `genkey` 时的密码 |

### 3. 正式 tag

```bash
git tag v0.8.0 && git push --tags
```

`v*` tag 触发 `.github/workflows/release.yml`（macOS amd64 DMG + Windows amd64 EXE + sign + publish）。

### 4. 发版记录

本文件 `docs/releases/v0.8.0.md` 作为 release notes 来源（`scripts/release.sh` 调用 `--notes docs/releases/v0.8.0.md`）。发版前补充以下内容：

- [ ] 实测的 updater 版本号（填入 TL;DR 的 vX.Y.Z）
- [ ] 真实的 `PublicKeyB64` 替换后的 commit SHA
- [ ] Secrets 配置完成的验证截图（可选）

## 红线（不做）

- ❌ 不做 Linux 客户端（用户拍板 v0.8.0 只 Windows + macOS）
- ❌ 不做 canary 多通道（stable 单通道）
- ❌ 不做 Windows Authenticode / macOS Notarization（私钥流程，v0.9+ 规划）
- ❌ 不做差量级热更新（block-level patch）
- ❌ 不做 Electron `electron-updater`（项目是 Wails + Go binary）
- ❌ 不引入 `minisign` / `go-selfupdate` 新依赖（Go 标准库 `crypto/ed25519` 够用）
