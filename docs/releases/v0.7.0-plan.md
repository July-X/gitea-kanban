### 关键设计决策

**D1：ListLabels 颜色字段处理**
- Gitea `LabelDTO.Color` 必返 hex（如 `#f29513`）
- GitHub `GET /repos/{owner}/{repo}/labels` 返 `{name, color, default, description}` —— **也**有 color 字段
- 差异：GitHub color 默认可能为 `ededed` 灰，与 Gitea 不一致；直接透传即可，前端按 hex 渲染不需适配
- **决策**：直接透传，**不**做兼容处理

**D2：UpdatePullMilestone GitHub 用 number vs Gitea 用 name**
- Gitea `milestone` 字段接受 string（title），返 string
- GitHub `milestone` 字段接受 number（id），返 number
- **决策**：DTO 字段类型不变（保持 string 语义），GitHub adapter 内部做 name → number 反查（ListMilestones 预加载建立 map）；前端 store 仍按 name 显示
- **影响**：`UpdatePullMilestoneArgs` 字段名/类型不变，新增 GitHub name→number 转换在 adapter 内部

**D3：ListBranches GitHub 数据源下不展示分支列表（用户拍板）**

> **背景**：v0.7 plan 初版把 `ListBranches` 列为 Phase 1.6 任务（建议走 go-git 本地 ref list）。用户拍板：GitHub 数据源下**不**做分支列表展示，理由：
>
> 1. **分支信息已经走 Git Graph**：所有分支的 commit DAG、ref 关系、branch tip 都在 Git Graph 中完整展示（v0.5.0 后 `TimelineNewView` 已实现 branch lane 渲染 + branch tip dot）
> 2. **SettingsView 那个分支列表是 Gitea 端增强体验**：用户接 Gitea 账号时列分支用于 PR 过滤、starred 分支管理；GitHub 端用户**没有**这个诉求（GitHub 用户主要看 PR，不直接管分支）
> 3. **避免 N×API 调**：GitHub 分支列表 API 在 monorepo 上可能返 1000+ 分支，rate limit 风险
>
> **决策**：
> - GitHub `ListBranches` 仍返 `ErrNotSupported`（保持现状）
> - 前端 `branch store` 在 GitHub 数据源下跳过加载（store action 内部 `if (platform === 'github') return []`）
> - SettingsView 入口在 GitHub 数据源下隐藏（与 v0.6.0 Issue 入口隐藏风格一致）
> - **后续**（v0.7+）：如需 GitHub 分支列表，可走 `go-git` 本地 ref list，单独开 task
>
> **红线条目**：v0.7.0 Plan 初版 D3（"ListBranches 走 go-git 本地 ref"）作废
>
> **D3 简化版**：GitHub 端 `ListBranches` 维持 `ErrNotSupported`；前端 store + view 在 GitHub 数据源下跳过；不引入新平台派发代码
