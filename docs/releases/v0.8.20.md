# v0.8.20 — 修复 darwin-amd64 用户下载更新 / manualOnly 误报

> 发布日期：2026-07-23
> commit：待 push 后回填

## 问题背景

用户报告：在 macOS amd64 上点击设置页"下载更新"，下载进度条一闪而过，最终无法自动下载。

日志证据：`22:04:09.520 update available current=master latest=v0.8.19 manualOnly=true manualReason="当前平台 darwin-amd64 没有预编译安装包"`

但实测 GitHub release v0.8.19 资产中**实际存在** `gitea-kanban-v0.8.19-macos-amd64.dmg`（10.3 MB）——后端根本没识别。

## 改动内容

### 1. extractPlatformFromAssetName 加 macos- → darwin- 归一化

**文件**：`app/updater/updater.go`

v0.8.0 拍板命名约定是 `gitea-kanban-vX.Y.Z-darwin-amd64.dmg`（`manifest.go:43` `PlatformKey` 的输出），但 `release.yml:311` 实际产物名是 `gitea-kanban-vX.Y.Z-macos-amd64.dmg`（macOS 命名习惯）。两者**从未对接**，导致 darwin-amd64 用户的 `Assets` map 一直空，`Check()` 永远返 `manualOnly=true`。

抽取 asset name 时加一次 `strings.Replace(afterVer, "macos-", "darwin-", 1)`，把资产名归一化成下游一致的 `darwin-*` key。

### 2. 补单元测试

**文件**：`app/updater/updater_test.go`

`TestExtractPlatformFromAssetName` 加 3 个用例：
- `gitea-kanban-v0.8.20-macos-amd64.dmg` → `darwin-amd64`
- `gitea-kanban-v0.8.20-macos-arm64.dmg` → `darwin-arm64`
- `gitea-kanban-v0.8.20-macos-universal.dmg` → `darwin-universal`

## 验证

- `go test ./app/updater/...` 全过（新增 3 测试用例）
- `pnpm typecheck` ✅（纯后端改动，前端不受影响）
- 手动验证：darwin-amd64 binary 跑 → 设置页点检查更新 → latest=v0.8.19 → 按钮变"下载更新"（不再是"前往下载页"）→ 点 → 进度条真实跑 → 落盘 dmg → Install 成功

## 影响范围

- 修所有 macOS amd64 用户的下载更新路径
- 不影响 macOS arm64 / windows / Linux / dev build
- 不动 Wails binding 契约、不动 DTO、不动 CI workflows

## 向后兼容

- 老版本 darwin-arm64 / darwin-universal 用户直接命中对应 key，不走归一化
- 归一化只影响 asset name 到 platform key 的映射，不修改任何对外接口

## 不动 / 不要做

- 不要改 `release.yml` 命名（要历史 release 重打，不在 fix 范围）
- 不要改 Wails bindings 契约
- 不要启用 darwin-universal → darwin-amd64 fallback（user 拍板：保守方案，避免无意中让用户装 universal 拖体积）
